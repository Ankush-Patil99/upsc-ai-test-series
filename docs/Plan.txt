SECTION 1: SHOULD YOU USE AN EDUCATIONAL/UPSC-SPECIALIZED LLM?
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

SHORT ANSWER: YES — but not off-the-shelf. You need a general-purpose
reasoning model that you fine-tune or prompt-engineer for UPSC.

WHY a dedicated educational LLM beats a generic one for this use case:

  1. UPSC MCQ Quality:
     - Generic LLMs hallucinate UPSC-specific facts (acts, articles, treaties)
     - A UPSC-tuned model learns the trickiness, distractors, and style of PYQs
     - Your RAG pipeline (PGVector context) becomes significantly more effective
       when the LLM knows how to USE the retrieved context for UPSC-style answers

  2. Instruction Following:
     - Your prompts require strict JSON output with keys:
       question, options[], correct, explanation, mains_facts
     - A well-fine-tuned model follows this reliably; generic 8B models often
       wrap output in markdown, miss keys, or produce malformed JSON

  3. Local = No Rate Limits:
     - Eliminates the 30-second sleep-and-retry cascade entirely
     - Generate 50+ MCQs in a batch without hitting API quotas
     - Full control — stop paying per token

PHASE 3: UPGRADE LANGGRAPH GRAPH ARCHITECTURE                       [Day 2-3]
═══════════════════════════════════════════════════════════════════════════════

  Current graph: retrieve → draft → critique → [loop or end]
  Problem: The critique node is FAKE (hardcoded, not a real LLM judge)

  3.1  Make the critique node a REAL LLM-as-judge node:
       - Feed draft_mcq + context to the local LLM
       - Ask it: "Does this MCQ have any hallucinations? Output JSON: {pass: bool, critique: str}"
       - Only loop if pass == false AND iterations < 3

  3.2  Add a "validator" node after critique:
       - Checks that all 4 options are distinct
       - Checks that "correct" is one of the 4 options
       - Checks mains_facts is < 50 words
       - If validation fails → regenerate

  3.3  Add a "difficulty calibrator" node:
       - After draft, check if the question difficulty matches requested level
       - Easy: direct factual recall | Medium: 2-statement elimination |
         Hard: multi-concept integration like actual UPSC PYQs

  3.4  Add LangGraph checkpointing (persistence):
       from langgraph.checkpoint.sqlite import SqliteSaver
       checkpointer = SqliteSaver.from_conn_string("checkpoints.db")
       graph.compile(checkpointer=checkpointer)
       - This allows resuming interrupted batch generations
       - Critical for production stability

  3.5  Add async support to batch_generator.py:
       - Use asyncio + astream_events from LangGraph for streaming MCQ generation
       - Send partial results to frontend as they complete

═══════════════════════════════════════════════════════════════════════════════

  6.1  Enable Ollama GPU acceleration:
       - Verify: ollama run qwen2.5:14b --verbose (check "using GPU" in output)
       - Set env var: OLLAMA_NUM_GPU=1 (or number of GPUs you have)

  6.2  Tune Ollama for throughput:
       Set in Ollama environment:
         OLLAMA_NUM_PARALLEL=4   → allow 4 concurrent requests
         OLLAMA_MAX_LOADED_MODELS=2  → keep model hot in VRAM
         OLLAMA_FLASH_ATTENTION=1    → faster attention, less VRAM

  6.3  Batch generation strategy:
       - Use Python asyncio.gather() to send multiple LangGraph invocations
         in parallel (each topic as a separate async task)
       - Cap concurrency at 4–6 for a single GPU to avoid OOM errors

  6.4  Add response caching:
       - Cache MCQs per (topic, difficulty) key in Redis or SQLite
       - Return cached MCQ if same topic+difficulty requested within 24h
       - This reduces GPU load for repeated topics dramatically
