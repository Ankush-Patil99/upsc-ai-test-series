????????????????????????????????????????????????????????????????????????????
1. SYSTEM ARCHITECTURE Ś TWO PHASES
????????????????????????????????????????????????????????????????????????????

PHASE 1 Ś Full Pipeline (FULL_PIPELINE_FOR_GITHUB/)
  6-node LangGraph state machine:
    Controller ? Researcher ? Drafter ? Engineer ? Formatter ? Orchestrator
  LLM: Cloud cascade across 5 providers with automatic failover:
    Together AI ? OpenRouter ? Groq ? Cohere ? Google Gemini (emergency)
  Vector Store: ChromaDB (primary) + PostgreSQL/pgvector (secondary)
  Embedding: BAAI/bge-large-en-v1.5 (primary), bge-small-en (dedup)

PHASE 2 Ś Production App (api/, src/, frontend/)
  3-node self-correcting LangGraph RAG pipeline:
    Retrieve ? Draft ? Critique (max 3 hallucination-correction loops)
  LLM: Llama 3.1 8B via Ollama (local inference on GPU)
  Vector Store: PostgreSQL + pgvector exclusively
  Embedding: sentence-transformers/all-MiniLM-L6-v2 (384-dimensional)

????????????????????????????????????????????????????????????????????????????
2. WHY WE MOVED FROM CLOUD CASCADE TO LOCAL INFERENCE
????????????????????????????????????????????????????????????????????????????

Problems encountered with the cloud cascade in Phase 1:
  [x] Groq free tier: 6,000 tokens/min cap Ś batch generation hit 429 errors
      repeatedly, forcing 30-second sleep-and-retry loops
  [x] Together AI / OpenRouter: per-day quotas broke the 24-hour APScheduler
      pipeline mid-run
  [x] Every MCQ generation required live internet access
  [x] API costs would scale badly with any meaningful batch size

Decision: Shifted to local Llama 3.1 8B via Ollama for Phase 2.
  [+] Zero token limits Ś 24hr scheduled pipeline runs uninterrupted
  [+] Zero API cost after initial model download
  [+] Consistent outputs at temperature=0.3
  [+] Fully offline after setup

Trade-off: ~30s local latency vs ~1-2s on Groq.
Acceptable because MCQs are pre-generated and cached Ś users see zero wait.
Production path: swap to Groq (GROQ_API_KEY already wired in .env).

????????????????????????????????????????????????????????????????????????????
3. WHY THE 6-NODE PIPELINE WAS SIMPLIFIED TO 3 NODES
????????????????????????????????????????????????????????????????????????????

  Classifier Node (in-graph) ? Moved to ingestion pipeline (classifier.py)
    Two sequential 8B LLM calls caused OOM errors on 8GB VRAM under load.

  Difficulty Calibration Node ? Removed
    Three sequential calls (Draft + Critique + Calibrate) exhausted VRAM.

  Multi-Model Cascade Node ? Removed
    Routing to 70B required disk swap (~45s) Ś unusable for real-time use.

  Separate Embedding Node ? Merged into ingestion pipeline
    Redundant once pre-ingestion was solidified; added latency with no gain.

  Result: Ingestion pipeline handles classification + embedding on a schedule.
  Real-time graph stays lean at 3 nodes, executable on a single consumer GPU.

4. RETRIEVAL & EMBEDDING (VERIFIED FROM CODE)
????????????????????????????????????????????????????????????????????????????

  Model         : sentence-transformers/all-MiniLM-L6-v2
  Dimensions    : 384
  Vector Store  : PostgreSQL + pgvector
  Chunk Size    : 1200 characters
  Overlap       : 200 characters
  Splitter      : RecursiveCharacterTextSplitter
  Dedup         : L2 distance < 0.15 Ś skip insert if near-duplicate found
  Retrieval k   : k=4 textbook facts + k=3 PYQ patterns
